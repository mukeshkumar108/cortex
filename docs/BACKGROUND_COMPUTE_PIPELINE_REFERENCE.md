# Synapse Background Compute Pipeline Reference

This document describes what Synapse computes in the background (and on startup brief assembly), with exact source paths, models, prompts, schedules, storage targets, and runtime consumers.

Scope: current implementation in `src/main.py`, `src/session.py`, `src/loops.py`, `src/openrouter_client.py`, `src/config.py`.

## 1) Scheduler and Cadence (what runs automatically)

Configured in `src/main.py` lifespan startup and `src/config.py` defaults.

- `user_model_updater_loop`
  - Enabled by default: `user_model_updater_enabled=true`
  - Interval default: every 300s
  - Purpose: fast heuristic updates to `user_model` from recent transcript + loops
- `user_model_enrichment_loop`
  - Enabled by default: `user_model_enrichment_enabled=false`
  - Interval default: every 900s
  - Purpose: LLM-based structured enrichment + narrative refresh + hygiene
- `loop_staleness_janitor_loop`
  - Enabled by default: `loop_staleness_janitor_enabled=true`
  - Interval default: every 86400s
  - Purpose: age-based loop status transitions (`active` -> `stale`/`needs_review`)
- `daily_analysis_loop`
  - Enabled by default: `daily_analysis_enabled=false`
  - Interval default: every 86400s
  - Purpose: compute prior-day themes/scores/steering in `daily_analysis`
- `daily_habit_dedupe_loop`
  - Runs alongside daily analysis loop (same interval)
  - Purpose: dedupe/flag uncertain daily habit loops
- `derived_memory_audit_loop`
  - Enabled by default: `derived_pipeline_audit_enabled=false`
- `proactive_shadow_candidates_loop`
  - Enabled by default: `proactive_shadow_candidates_enabled=false`
- `v2_invariant_checker_loop`
  - Enabled by default: `v2_invariant_checker_enabled=false`
- Optional ops loops
  - `idle_close_loop` (default disabled)
  - `drain_loop` outbox drainer (worker default enabled in compose)

Runtime ownership:

- `api` runtime does not start broad background compute.
- `worker` runtime owns the full loop set.
- If the API is misconfigured with loops enabled, only `idle_close` and `outbox_drain` are allowed to start.
- LLM-heavy loops (`user_model_enrichment`, `daily_analysis`, `daily_habit_dedupe`) apply a startup delay via `LLM_BACKGROUND_LOOPS_STARTUP_DELAY_SECONDS` before the first run.

## 2) Session Ingest and Outbox Jobs

Primary flow is enqueue-first and durable.

### 2.1 Ingest job creation

- Entry: `session.enqueue_session_ingest(...)` in `src/session.py`
- Writes full messages to `session_transcript`
- Enqueues outbox job:
  - `job_type=session_raw_episode`
  - `dedupe_key=session_ingest_raw:{tenant}:{user}:{session}`
  - table: `graphiti_outbox`

### 2.2 Raw episode processing

- Handler: `_handle_session_raw_episode_job` in `src/session.py`
- Reads transcript back from `session_transcript`
- Caps message count/chars for safety
- Sends to Graphiti via `add_session_episode(...)`
- On success, enqueues post-ingest hooks:
  - `session_summary`
  - `open_loops`

### 2.3 Failure semantics

- Permanent/validation failures can dead-letter to `graphiti_outbox.status='failed'`
- Example observed failure: missing transcript messages causes `validation error: missing transcript messages for session raw episode`
- If this job fails, downstream summary/loops hooks are not produced for that session

## 3) Session Summary Computation

### 3.1 When generated

- Generated by post-ingest hook `session_summary` (`_execute_post_ingest_hook` in `src/main.py`)
- Input: full/capped transcript for that session

### 3.2 Model and prompt

- Function: `_summarize_session_close(...)` in `src/session.py`
- LLM task: `task="session_episode"`
- Model route: `openrouter_model_session_episode` (default `xiaomi/mimo-v2-flash`)
- Prompt is strict structured recap with sections:
  - `SUMMARY`, `TONE`, `DECISIONS`, `UNRESOLVED`, `MOMENT`, `BRIDGE`
  - hard constraints on tense, max words, no coaching, no hallucination

### 3.3 Quality layers

In `summarize_session_messages_with_quality(...)`:

- Primary LLM output (`llm_primary`)
- Repair pass for summary/bridge if malformed (`llm_repair`)
- Deterministic fallback if still invalid (`deterministic_fallback`)

Stored quality metadata includes:
- `summary_quality_tier`
- `summary_source`

### 3.4 Where stored and used

- Stored to Graphiti `SessionSummary` node via `graphiti_client.add_session_summary(...)`
- Extra attributes persisted: `summary_facts`, `tone`, `moment`, `decisions`, `unresolved`, `index_text`, `salience`
- Used by:
  - `/session/startbrief` evidence selection/ranking
  - daily analysis (preferred input path)
  - user model enrichment indexing/ranking context

## 4) Loop Extraction and Maintenance

### 4.1 Loop extraction compute

- Trigger: post-ingest hook `open_loops`
- Function: `loops.extract_and_create_loops(...)`
- Model task: `task="loops"`
- Model route: `openrouter_model_loops` (default `xiaomi/mimo-v2-flash`)
- Prompt enforces strict JSON and low-noise extraction rules (commitment/habit/friction/thread/decision gates)

### 4.2 Loop storage

- Table: `loops`
- Fields include `type`, `status`, `text`, `salience`, `time_horizon`, metadata/domain/reason, timestamps

### 4.3 Loop aging janitor

- Function: `apply_global_staleness_policy()` in `src/loops.py`
- Rules:
  - `today` older than 48h -> `stale`
  - `this_week` older than 10d -> `stale`
  - `ongoing` older than 21d -> `needs_review`

### 4.4 Daily habit dedupe

- Function: `_run_habit_dedupe_for_user(...)` in `src/main.py`
- Uses LLM (`task="loops"`) to produce:
  - merge groups (duplicate habits)
  - uncertain habits for confirmation
- Applies status/metadata updates in `loops`

## 5) User Model: Fast Updater (heuristic, non-LLM)

### 5.1 When

- `user_model_updater_loop` every 300s by default
- Selects users with recent transcript/loop activity in lookback window

### 5.2 How computed

- Function: `_propose_user_model_patch(...)`
- Uses regex/pattern extraction from recent user turns + active loops
- Extracts/updates candidates for:
  - `north_star` (vision/goal)
  - `current_focus`
  - `work_context`
  - `key_relationships`
  - `patterns`
  - `preferences`
  - optional health/spirituality anchors from loop domain

### 5.3 Storage and merge

- Upsert target: `user_model` table
- Merge logic via `_apply_user_model_proposal(...)`
- Source marker often `auto_updater`

### 5.4 Important caveat

- This path is regex-based and can introduce bad structured facts if extraction pattern is wrong.
- Example known risk: relationship regex can mis-parse relation/name pairs.

## 6) User Model: Enrichment Loop (LLM)

### 6.1 When

- `user_model_enrichment_loop` every 900s by default
- Runs two modes each cycle:
  - `daily` candidate set (recent activity)
  - `weekly` candidate set (broader recency)
- Has retry/backoff via `user_model_enrichment_state`

### 6.2 Evidence inputs

- Pulls recent Graphiti session summaries for indexing/ranking
- Pulls raw transcript user turns as actual evidence payload
- Uses configurable caps by mode (sessions, turns, chars)

### 6.3 Model and prompts

- Task: `user_model_enrichment`
- Model route: `openrouter_model_summary` (default `amazon/nova-micro-v1`)
- First prompt: structured profile enrichment JSON with required evidence objects
- Second prompt: narrative refresh JSON:
  - `narrative_stable`
  - `narrative_current`
  - with recency/contradiction handling rules

### 6.4 Storage and state

- Merged into `user_model`
- State tracked in `user_model_enrichment_state`:
  - last daily/weekly enrichment timestamps
  - retry counters / next retry / last error

### 6.5 Hygiene pass

- Also runs periodic hygiene cleanup (~20h guard)
- Sanitizes malformed relationships/patterns and migrates preference notes into daily anchors

## 7) Daily Analysis (themes, scores, steering)

### 7.1 When

- `daily_analysis_loop` every 86400s
- Target date defaults to yesterday (`daily_analysis_target_offset_days=1`)

### 7.2 Evidence selection

Preferred path:
- Pull session summaries from Graphiti for target day (`_get_user_daily_session_summaries`)
- Trim by salience and prompt char budget

Fallback path:
- Use raw transcript turns from `session_transcript` for target day

### 7.3 Model and prompt

- Function: `_generate_daily_analysis(...)`
- Task: `daily_analysis`
- Model route: `openrouter_model_summary` (default `amazon/nova-micro-v1`)
- Prompt requires strict JSON:
  - `themes` (psychological/emotional patterns)
  - `scores` (curiosity/warmth/usefulness/forward_motion 1-5)
  - `steering_note` (single concrete sentence)
  - `confidence`
- Includes anti-generic rejection and one regeneration attempt

### 7.4 Fallback analyzer

If LLM parsing/rejection fails, `_fallback_daily_analysis(...)` is used:
- keyword/regex pattern detection
- fixed steering sentence template
- `source="fallback"`, low confidence

### 7.5 Storage and usage

- Table: `daily_analysis`
- metadata includes:
  - `input_mode` (`session_summaries` vs `fallback_raw_turns`)
  - `session_count`, `turn_count`, `session_ids_used`
  - salience counts, analysis version, quality flags
- Consumed by `/session/startbrief` as:
  - `ops_context.yesterday_themes`
  - `ops_context.steering_note`
  - `ops_context.daily_analysis` freshness fields

## 8) Startbrief Assembly (on demand, not cron)

Endpoint: `GET /session/startbrief` in `src/main.py`.

It computes a runtime package from multiple stores:

- Time context
  - local time label, gap minutes, sessions today
- Session evidence
  - latest + recent `SessionSummary` nodes from Graphiti
  - fallback to legacy episode summaries if needed
- Loop evidence
  - top loops from `loops` table
- Yesterday analysis
  - from `daily_analysis` table for `reference_date - 1 day`
- User model hints
  - from high-confidence sections of `user_model`
- Entity hints / profiles
  - from Graphiti node search + relationship entities from `user_model` + summary recurrence fallback

### 8.1 `handover_text` generation

- Prompt function: `_generate_startbrief_bridge_llm(...)`
- Task: `startbrief_bridge`
- Model route: `openrouter_model_summary` (default `amazon/nova-micro-v1`)
- Inputs include selected thread text, loops, yesterday themes, steering note, continuation hint, current narrative
- Then validated/compressed/sanitized; if invalid -> fallback deterministic handover text

### 8.2 `narrative`

- Built by `_build_startbrief_narrative(...)`
- Sources: `user_model` narratives + selected summary/loop evidence

### 8.3 `ops_context`

Built by `compose_ops_context(...)` and includes:
- `top_loops_today`
- `user_model_hints` + `user_model_hint_rows` (with confidence/updated_at/source)
- `yesterday_themes`
- `steering_note`
- `daily_analysis` freshness tuple (`date`, `confidence`, `updated_at`)

### 8.4 Startbrief provenance and freshness

- Response includes `evidence` block:
  - summary IDs used/fetched
  - ranking breakdown
  - fallback flags
  - daily analysis date used
  - ingest freshness from outbox backlog status
- Full response context logged to `startbrief_history`

## 9) Entity Hint Package (how entities are produced)

Entity hints for startbrief are assembled by `_build_entity_candidates(...)` plus fallback logic.

Sources mixed together:
- Relationship entities extracted from `user_model.key_relationships`
- Graphiti `search_nodes` with queries built from context summaries + top loops + relationship seeds
- Summary-text recurrence fallback (`_extract_hints_from_summary_texts`) when hint count is low

Entity profiles (for key relationships) are built from Graphiti `search_facts` queries per relationship name.

Important: this is currently a hybrid runtime builder, not a dedicated offline entity graph curation pipeline.

## 10) Where each computed artifact is consumed

- `session_transcript`
  - source of truth for raw session messages
  - consumed by ingest hooks, daily analysis fallback, enrichment transcript evidence
- Graphiti `SessionSummary` nodes
  - consumed by startbrief ranking, daily analysis preferred input, enrichment index/ranking
- `loops`
  - consumed by startbrief `top_loops_today`, user-model updater fallback signals, entity ranking context
- `user_model`
  - consumed by startbrief narrative + hints + relationship/entity role enrichment
- `daily_analysis`
  - consumed by startbrief yesterday themes/steering/freshness
- `startbrief_history`
  - debugging/provenance for what startbrief actually used
- `graphiti_outbox`
  - durability/freshness signal (backlog => stale startbrief risk)

## 11) Known stale-data and mismatch failure modes

1. Missing ingest for a session
- If `session_raw_episode` fails, no summary hook/open-loops hook runs for that session.
- Downstream daily analysis may miss that session in summary mode.

2. Daily analysis fallback path
- If LLM output is rejected/invalid, fallback analyzer can emit generic/stale-feeling steering text.

3. User model updater regex drift
- Heuristic extraction may inject wrong relationship/work hints and keep them until replaced.

4. High-confidence gating + freshness filtering mismatch
- Startbrief only surfaces user-model hints above confidence thresholds and applies age filtering in ingredient selection.
- Old but high-confidence fields can still leak if refresh logic never corrected them.

5. Entity hint hybrid sourcing
- Runtime hint builder mixes graph search + summary recurrence + user_model roles; quality depends on all upstream artifacts being healthy.

## 12) Model routing summary (single view)

From `src/openrouter_client.py` task router:

- `summary` -> `openrouter_model_summary` (default `amazon/nova-micro-v1`)
- `loops` -> `openrouter_model_loops` (default `xiaomi/mimo-v2-flash`)
- `session_episode` -> `openrouter_model_session_episode` (default `xiaomi/mimo-v2-flash`)
- `identity` -> `openrouter_model_identity` (default `amazon/nova-micro-v1`)
- `daily_analysis` -> `openrouter_model_summary`
- `startbrief_bridge` -> `openrouter_model_summary`
- `user_model_enrichment` -> `openrouter_model_summary`

Note: `_summarize_session_bridge(...)` calls `task="session_bridge"`, which is not an explicit router branch, so it falls to fallback model selection behavior in `_call_llm`.

## 13) Practical debugging checklist (when outputs look wrong)

1. Check outbox health first (`graphiti_outbox`) for failed/pending `session_raw_episode` and hook jobs.
2. Check whether target-day sessions have corresponding `SessionSummary` nodes.
3. Check `daily_analysis.metadata.input_mode` and `quality_flag` for fallback indicators.
4. Check `user_model` field-level `updated_at`, `source`, `confidence` inside sections used for hints.
5. Use startbrief provenance (`startbrief_history` and `evidence` block) to verify what was actually used in that response.
